## Deuda técnica y correcciones pendientes

### 🔴 Crítico

#### `POST /api/subscription` ignora el plan enviado
El frontend (`handleUpgrade(planKey)`) envía `{ plan: planKey }` en el body, pero el backend lo ignora por completo y siempre crea una suscripción PRO en Mercado Pago. Si se crean planes BASIC, PREMIUM o ENTERPRISE en el admin, el checkout va a llevar siempre al precio de PRO.

**Corrección necesaria:**
```typescript
// src/app/api/subscription/route.ts
export async function POST(request: Request) {
  const body = await request.json();
  const planKey = body.plan ?? "PRO"; // leer el plan del body
  // buscar el plan en DB para obtener el precio y el preapproval_plan_id de MP
  const planConfig = await prisma.planConfig.findFirst({ where: { plan: planKey } });
  // usar planConfig.price y el plan_id configurado para crear el checkout
}
```

#### `PLANS` en `src/lib/mercadopago.ts` está hardcodeado
El objeto `PLANS` define FREE y PRO con valores fijos. Si el admin edita el precio de un plan en la DB, `GET /api/subscription` seguirá devolviendo el precio viejo del hardcode para construir el `planInfo` del usuario. Esto genera inconsistencia entre lo que se muestra en el dashboard y lo que se cobra realmente.

**Corrección necesaria:** En `GET /api/subscription`, obtener el `planInfo` desde `prisma.planConfig` en lugar de `PLANS[plan]`.

### 🟡 Importante

#### `PLAN_ORDER` duplicado entre frontend y backend
El objeto `{ FREE: 0, BASIC: 1, PRO: 2, PREMIUM: 3, ENTERPRISE: 4 }` existe en `subscription-card.tsx` como constante client-side. Si se agrega un plan nuevo en el admin con una clave distinta (ej: `STARTER`), no va a aparecer en la lógica de upgrade.

**Alternativa más robusta:** Usar directamente `sortOrder` de la DB para comparar, sin depender de un mapa estático. El endpoint `/api/plans` ya devuelve `sortOrder`, usarlo como fuente de verdad:
```typescript
const currentSortOrder = allPlans.find(p => p.plan === currentPlanKey)?.sortOrder ?? -1;
const upgradePlans = allPlans.filter(p => p.sortOrder > currentSortOrder);
```

#### El modo mantenimiento se chequea en `layout.tsx` (Server Component)
Esto funciona pero tiene una limitación: el chequeo depende del header `x-pathname` seteado por el middleware (`proxy.ts`). Si en algún momento el middleware no propaga ese header (ej: rutas de assets, error boundaries), el chequeo falla silenciosamente y no redirige.

**Sugerencia:** Mover la lógica de mantenimiento al middleware (`proxy.ts`) donde se tienen todas las rutas con seguridad, y eliminar el chequeo duplicado en `layout.tsx`.

#### `GET /api/subscription` devuelve `planInfo` desde `PLANS` hardcodeado
Consecuencia directa del punto anterior en 🔴. El campo `price` en la respuesta de `GET /api/subscription` no refleja cambios hechos desde el admin de planes.

### 🟢 Mejoras deseables (sin urgencia)

#### Preapproval Plan IDs de Mercado Pago por plan
Para soportar múltiples planes de pago en MP, cada `PlanConfig` en la DB necesita un campo `mpPlanId` (el ID del `preapproval_plan` creado en el panel de MP). Hoy solo existe el plan PRO configurado en MP.

**Schema sugerido:**
```prisma
model PlanConfig {
  // ... campos existentes
  mpPlanId  String? // ID del plan en Mercado Pago
}
```

#### Rate limiting en `GET /api/plans`
El endpoint es público y sin auth. Considerar agregar rate limiting (ya existe `src/lib/rate-limit.ts`) para evitar abuso.

#### Caché en `GET /api/plans`
Los planes cambian raramente. Agregar `revalidate` o headers de caché evitaría hits innecesarios a la DB en cada visita a la landing:
```typescript
// src/app/api/plans/route.ts
export const revalidate = 3600; // 1 hora
```

#### Feedback visual al reordenar planes en el admin
Actualmente los botones ↑↓ envían el PATCH y actualizan el estado local. Si el request falla, el estado local queda desincronizado con la DB sin ningún aviso. Agregar un `toast.error` en el catch del `handleMove`.

#### Confirmar antes de eliminar un plan en el admin
No hay un dialog de confirmación al borrar un plan. Si hay usuarios activos en ese plan, eliminarlos puede generar inconsistencias.
